# Discasa v0.1 Architecture

## Core principle

Discord is the storage transport and canonical remote backend.
The local app keeps a searchable cache and metadata index for a better UX.

## Components

### Desktop app

Responsibilities:
- Login trigger
- Session status
- Guild selection
- Initialization of Discasa in a chosen guild
- Drag and drop upload
- Collection management
- Local cache/index

### Server app

Responsibilities:
- Discord OAuth2 authorization code flow
- Session/token exchange and refresh
- Bot-backed setup inside the guild
- Upload orchestration
- Validation of guild eligibility

### Shared package

Responsibilities:
- DTOs
- Constants
- Shared validation helpers

## Discord structure

Category:
- `Discasa`

Channels:
- `discasa-drive`
- `discasa-index`
- `discasa-trash`

## Metadata strategy

Collections should remain local/app-level metadata rather than Discord channels.
Every uploaded file should be linked to:
- guild id
- drive channel id
- message id
- attachment url
- collection ids
- timestamps

## v0.1 implementation constraint

Keep the sync model simple:
- single active guild
- one-way upload flow first
- no cross-device reconciliation yet
- no real-time delete sync yet
